# 吉他乐理学习网站：Agent 开发交付规格说明书

> 文档用途：本文件用于直接交付给开发 Agent，作为产品、设计、前端、后端、数据库与部署实现依据。
>
> 当前阶段目标：优先完成可部署在 Cloudflare 免费额度内的 MVP。课程是主线，乐理实验室与练习系统帮助理解，歌曲分析负责真实应用。暂不开发“AI 乐理老师”。

---

## 1. 项目背景

很多吉他初学者只会机械记忆和弦形状、音阶框型和扫弦套路，却不知道：

- 为什么 C 和弦由 C、E、G 构成；
- 为什么某些和弦经常一起出现；
- 为什么同一个音在不同调中作用不同；
- 为什么五声音阶可以用于某些和弦进行；
- 如何从音符、音程、音阶、调性推导出和弦与和声；
- 如何把抽象乐理映射到真实吉他指板。

网上的吉他乐理资料通常存在以下问题：

1. 内容零散，缺少完整学习路径；
2. 只教指型，不解释原理；
3. 乐理知识与吉他指板割裂；
4. 课程、工具、练习和歌曲分析彼此独立；
5. 初学者看完教程后仍不会迁移到真实歌曲。

本项目希望建立一个系统性的吉他乐理学习网站，让用户完成：

> 形状记忆 → 理解音乐结构 → 在指板上观察 → 通过练习巩固 → 应用到真实歌曲

---

## 2. 产品定位

### 2.1 一句话定位

一个以系统课程为主线，将抽象乐理、吉他指板、练习和歌曲分析连接起来的交互式吉他乐理学习网站。

### 2.2 核心价值

- 不只告诉用户“怎么按”，还解释“为什么这样按”；
- 不只讲乐理定义，还展示其在吉他指板上的空间关系；
- 不只让用户看懂，还通过练习验证是否真正掌握；
- 不只分析理论，还把理论迁移到真实歌曲；
- 不依赖大模型生成基础乐理事实，所有核心计算由确定性规则完成。

### 2.3 核心学习闭环

```text
系统课程
   ↓
声音与指板演示
   ↓
交互式乐理实验
   ↓
即时练习与错题
   ↓
歌曲应用与扒谱
   ↓
复习与能力追踪
```

## 3. 目标用户

### 3.1 主要用户

- 完全零基础、刚开始学吉他的用户；
- 会弹开放和弦，但不懂和弦构成的用户；
- 会弹唱，但不理解调性和和弦级数的用户；
- 会背五声音阶框型，但不会连接指板和跟随和弦变化的用户；
- 想学习歌曲分析、即兴和基础创作的吉他手。

### 3.2 典型用户目标

- 系统学习基础乐理；
- 理解吉他指板；
- 理解和弦与音阶；
- 学会分析歌曲；
- 学会移调和使用变调夹；
- 为即兴和创作建立理论基础。

## 4. 产品原则

1. **课程优先**：课程是网站主线，不能把网站做成工具集合。
2. **先理解后探索**：用户先通过课程建立基本概念，再进入实验室自由探索。
3. **边学边练**：每节课包含音频、指板和即时练习。
4. **乐理与吉他绑定**：所有抽象概念尽量提供吉他指板映射。
5. **同一知识多种表示**：音名、唱名、级数、音程、指板位置相互切换。
6. **确定性计算优先**：和弦、音阶、调内和弦、罗马数字、移调等使用程序规则计算。
7. **自动扒谱是辅助工具**：产品名称与交互要体现“AI 辅助扒谱草稿”，不能承诺绝对准确。
8. **MVP 控制复杂度**：免费部署阶段不运行服务端 Python 音频模型、不使用 GPU、不运行常驻任务进程。
9. **内容可审计**：Agent 生成课程后必须支持草稿、审核、发布和版本记录。
10. **暂不开发 AI 乐理老师**：不提供聊天式教师入口。

## 7.2 课程目录页

采用三栏布局：


课程状态：

- `✓` 已掌握
- `●` 正在学习
- `○` 尚未学习
- `!` 建议复习
- `🔒` 前置课程未完成

## 7.3 单节课程页

采用三栏布局：

```text
左栏：本章目录
中栏：教学内容
右栏：本课公式、前置知识、相关课程
```

课程内容固定顺序：

1. 核心问题；
2. 声音演示；
3. 乐理解释；
4. 指板可视化；
5. 真实和弦或歌曲案例；
6. 即时练习；
7. 本课总结。

## 7.4 乐理实验室

实验室使用大画布布局：

```text

底部：提交按钮
```

## 8. 视觉规范

### 8.1 风格

采用：

> 温暖的音乐教室感 + 清晰的数字化工具感

避免：

- 全黑摇滚酒吧风；
- 高饱和儿童教育风；
- 过多渐变、霓虹和复杂阴影。

### 8.2 色彩

建议：

- 页面背景：暖白或浅米灰；
- 主色：深蓝、墨绿或深紫；
- 强调色：琴木棕或橙黄；
- 正确：绿色；
- 错误：红色；
- 根音：固定高辨识度颜色；
- 三音、五音、七音：使用同一套稳定配色。

Agent 必须建立颜色 token，不能在组件内随意写颜色。

### 8.3 字体层级

```text
页面标题：32–40px
章节标题：24–28px
卡片标题：18–20px
正文：16–18px
辅助说明：13–14px
```

### 8.4 组件风格

- 适中圆角；
- 轻微阴影；
- 主要靠留白分隔；
- 一张卡片只放一个主要动作；
- 主按钮实心，次按钮描边；
- 表格和时间轴强调可读性，不追求装饰。

### 8.5 响应式

- 桌面端三栏；
- 平板端两栏；
- 移动端单栏；
- 课程目录改为抽屉；
- 指板支持横屏；
- 扒谱深度编辑优先桌面和平板。

## 11. 确定性乐理规则引擎

基础乐理不能依赖大模型即时生成。需要实现独立的 TypeScript 规则库。

### 11.1 必须支持

- 音名与 MIDI 编号转换；
- 半音计算；
- 音程名称计算；
- 音阶生成；
- 和弦组成音生成；
- 和弦符号解析；
- 调内三和弦与七和弦生成；
- 罗马数字转换；
- 移调；
- 变调夹计算；
- 指板位置计算；
- CAGED 根音与和弦音标记；
- 和弦进行的基础调性候选；
- 调内、调外和弦判断。

### 11.2 示例接口

```ts
getScaleNotes("C", "major")
// ["C", "D", "E", "F", "G", "A", "B"]

getChordNotes("Cmaj7")
// ["C", "E", "G", "B"]

analyzeChordInKey("Em", "G", "major")
// { degree: "vi", diatonic: true, function: "tonic" }

transposeChord("C#m7", 2)
// "D#m7"
```

### 11.3 拼写要求

规则引擎需区分升降号语境，不能全部强制转换为升号。例如：

- F 大调使用 B♭，不使用 A#；
- E♭ 大调使用 E♭、A♭、B♭；
- 和弦与音阶输出应优先使用当前调的正确拼写。

## 13. 自动扒谱功能设计

## 13.1 产品定义

名称使用：

> AI 辅助扒谱工作台

不能使用：

> 一键精准自动扒谱

### 13.2 MVP 输入

- 本地 MP3；
- WAV；
- 浏览器录音；
- 用户确认拥有合法使用权的音频。

### 13.3 MVP 输出

- BPM 候选；
- 拍点；
- 调性候选；
- 基础和弦时间轴；
- 和弦置信度；
- 用户可编辑结果；
- 罗马数字；
- 调内/调外标记；
- 基础和声解释；
- 一键移调；
- 变调夹建议；
- 保存项目。

### 13.4 处理策略

Cloudflare 免费阶段，音频分析优先在浏览器端执行：

```text
用户选择音频
→ 浏览器解码
→ Web Audio API / Essentia.js 提取特征
→ 浏览器生成基础分析结果
→ 用户修正
→ 将 JSON 结果提交 Worker
→ 保存到 D1
```

MVP 可选择不上传原始音频，仅保存分析结果。用户主动选择“云端保存音频”时，再上传到 R2。

### 13.5 后期升级接口

未来增加外部 Python/GPU 服务时：

```text
Cloudflare Worker
→ 创建外部处理任务
→ Python 音频服务处理
→ 回调 Worker
→ Worker 写入 D1/R2
```

前端数据结构保持不变。

### 14.2 不在 MVP 使用

- Redis；
- Docker 后端；
- Python 常驻服务；
- GPU 服务；

## 18. 身份验证

Cloudflare MVP 可先采用以下两种方案之一：

### 方案 A：先不登录

- 用户进度保存 LocalStorage/IndexedDB；
- 扒谱项目保存在浏览器；
- 最适合最快上线；
- 后续再迁移到账号体系。

### 方案 B：轻量账号

- 使用第三方 Auth 服务；
- Worker 验证 JWT；
- D1 保存用户资料与进度。

建议第一阶段先实现游客模式，同时预留 `user_id` 数据字段。
